Laravel Safeguard

Fail-closed file-upload validation: blocks polyglot web shells, malicious PDFs/SVGs, zip bombs, Office macros, and spoofed MIME types — with structural parsing and content sanitization, not just regex.

PHP 8.1 – 8.5 Laravel 10 / 11 / 12 / 13 MIT Fail-closed

What it is

Laravel Safeguard is a set of validation rules that inspect the real contents of an uploaded file before your app stores it. Its governing principle is to fail closed: when it cannot be sure a file is safe, it blocks the upload. Unknown content types, unparsable containers, and scanner errors all resolve to reject for security checks — never to "allow".

🐚 Always-on code scan

Every upload is scanned for PHP/script openers regardless of type. A valid image/PDF/ZIP header never exempts a file, so polyglot web shells are caught.

🔍 Structural MIME detection

Classifies by byte structure, disambiguates OLE/ftyp/RIFF/ZIP families (real .xls → Excel, JAR/APK), and returns untrusted for unknown content.

🧼 SVG sanitization

SVGs run through an allowlist sanitizer; the cleaned SVG is what gets stored. Scripts, handlers, and DTD/XXE are removed.

📄 PDF decode-before-scan

Flate/LZW/ASCII85/ASCIIHex and object streams are inflated and #xx names decoded before matching /JavaScript, /OpenAction, …

📦 Real zip-bomb detection

Archives are streamed against a hard cap on actual decompressed bytes; forged sizes, traversal, ADS, symlinks, and nested archives are handled.

📝 Office macros

VBA/OLE/ActiveX in both OOXML and legacy OLE/CFB (.doc/.xls/.ppt), resolved via relationships and content types.

Installation

composer require abdian/laravel-upload-guard

The service provider auto-registers via package discovery. Publish the config to tune behavior:

php artisan vendor:publish --tag=safeguard-config

Requirements

  • PHP 8.1–8.5, Laravel 10 / 11 / 12 / 13
  • Required extensions: fileinfo, zip, dom, libxml, exif
  • Optional: gd or imagick (only for the image re-encode mode)

Quick start

Add the single safeguard rule to any file field:

public function upload(\Illuminate\Http\Request $request)
{
    $request->validate([
        'file' => 'required|safeguard',
    ]);

    // The file is safe to store.
    $path = $request->file('file')->store('uploads');
}

By default, that one rule runs: structural MIME detection + dangerous-type blocking, strict extension/content matching, always-on code scanning, SVG sanitization, image scanning, PDF scanning, and archive + Office macro scanning — no extra calls needed.

Pairing with Laravel's mimes

Combine with Laravel's native mimes rule and Safeguard will enforce that the file's real content type matches the allowed extensions (defeats extension spoofing):

$request->validate([
    'avatar' => 'required|safeguard|mimes:jpg,png,webp',
    'doc'    => 'required|safeguard|mimes:pdf,docx',
]);

Fluent API

Use the rule object for fine-grained control:

use Abdian\UploadGuard\Rules\Safeguard;

$request->validate([
    // Images: type filter + dimensions + privacy
    'avatar' => ['required', (new Safeguard())
        ->imagesOnly()
        ->maxDimensions(1920, 1080)
        ->minDimensions(64, 64)
        ->blockGps()
    ],

    // PDFs: page limit + block active content
    'report' => ['required', (new Safeguard())
        ->pdfsOnly()
        ->maxPages(50)
        ->blockJavaScript()
        ->blockExternalLinks()
    ],

    // Documents (archive + macro scanning already on by default)
    'file' => ['required', (new Safeguard())->documentsOnly()],
]);

Individual rules

Each check is also available as a standalone string rule:

$request->validate([
    'avatar'   => 'required|safeguard_mime:image/jpeg,image/png|safeguard_image',
    'icon'     => 'required|safeguard_svg',
    'document' => 'required|safeguard_pdf|safeguard_pages:1,10',
    'archive'  => 'required|safeguard_archive',
    'report'   => 'required|safeguard_office',
]);

Common recipes

Avatar upload (image only, no GPS, re-encoded)

// config/safeguard.php → image_scanning.reencode = true  (strips any payload)
$request->validate([
    'avatar' => ['required', (new Safeguard())->imagesOnly()->maxDimensions(1024,1024)->blockGps()],
]);

User-supplied ZIP (block dangerous contents)

$request->validate(['bundle' => 'required|safeguard']); // archive scanning is on by default

Safely moving a validated upload (closes the TOCTOU window)

// The trait helper rejects traversal/symlink destinations before the move.
$dest = storage_path('app/uploads/' . $safeName);
// validateDestinationPath() lives on the ValidatesFileAccess trait used by scanners.

Configuration

Highlights from config/safeguard.php (fully commented in the published file):

'max_scan_size'   => 25 * 1024 * 1024,  // larger files are rejected…
'over_cap_policy' => 'reject',            // …or 'header_only'

'archive_scanning' => [
    'enabled' => true,                            // ON by default
    'max_decompressed_size' => 500 * 1024 * 1024, // hard cap on ACTUAL bytes
    'max_files_count' => 10000,
    'max_nesting_depth' => 3,
],

'office_scanning' => ['enabled' => true, 'block_macros' => true, 'block_activex' => true],

'svg_scanning' => ['mode' => 'sanitize'],   // or 'reject'

'image_scanning' => [
    'max_pixels' => 64_000_000,
    'reencode' => false,                         // recommended; needs ext-gd/imagick
],

'rate_limiting' => ['enabled' => false],     // DoS guard (opt-in)
'quarantine'    => ['enabled' => false],     // keep rejected files for forensics (opt-in)
Migrating from a pre-1.0 dev install? Several keys were removed/renamed (e.g. max_compression_ratio → max_decompressed_size). See UPGRADE.md for the full migration table and behavior changes.

Hardening notes

TOCTOU: scanning the PHP temp file does not by itself close the temp→storage time-of-check/time-of-use window. Use validateDestinationPath() immediately before moving a validated upload, and prefer enabling image_scanning.reencode.
SVG storage: in sanitize mode the uploaded file is rewritten in place with the cleaned SVG, so ->store() persists the safe version automatically.
Stricter than lax validators: Safeguard intentionally rejects polyglots, compressed/encrypted PDFs with active content, forged-size archives, legacy-macro Office docs, and DTD-bearing SVGs. Conversely it fixes common false positives (real .xls/.ppt, .js/.csv/.py text, compressed PDFs).

What it is NOT

Safeguard is a synchronous validation library, not an antivirus. It does not perform AV signature scanning, sandboxed detonation, or ML classification. It defends the documented threat classes; pair it with OS-level controls (non-executable upload dirs, least privilege) for defense in depth.

FAQ

A legitimate file is being rejected — what do I do?

Check the validation message. Common causes: the file's real type doesn't match its extension (rename it correctly), it exceeds max_scan_size, or it genuinely contains a script opener / macro. You can relax specific checks via config (e.g. svg_scanning.mode, office_scanning.block_macros) or fluent calls.

Does it work in queues / CLI (no HTTP request)?

Yes. Scanners guard config()/request()/logger() access and degrade safely without a bound request.

How do I report a security issue?
